Assetra.Application 架構缺點評估

選擇你最想優先解決的問題（可多選）

A

ViewModel 注入 15 個服務

建構子過長，PortfolioViewModel 職責過廣，難以測試與維護

B

Pass-through 服務沒有附加價值

AlertQueryService、LoanScheduleQueryService 只包一層 repo，增加複雜度卻無實質好處

C

執行責任不一致

TransactionWorkflowService 回傳 Plan（ViewModel 再持久化）vs SellWorkflowService 自己持久化，兩種模式混用

D

Namespace 不一致

專案資料夾 Assetra.Application，但 namespace 是 Assetra.AppLayer

E

Anemic Domain Model 風險

Core Model 逐漸變成純資料容器，計算邏輯全移至 Application 層，領域知識碎片化
